# biolab-quality-journal

Веб-застосунок для ведення журналу варок, журналу готової продукції і
автоматичної генерації паспортів якості (CoA) на партію продукції Biolab Group.

Замінює крихкий Google Sheet
`журнал обліку і контролю якості_Bulk_ГП` (ID `1aZl3uuEYQm2VGF1-6w7RF_IaubOpHS7gkrmxUZc2bE8`),
у якому формули рвуться від ручного редагування 2-5 одночасних користувачів.

## Мета

1. **Заводити специфікації** на продукт (RP-код → норми pH/в'язкість/мікробіологія,
   PAO, термін придатності, статус "затверджено/потребує верифікації")
2. **Реєструвати варки** (bulk партії) з фіз-хім показниками; висновок OK/OUT
   рахується автоматично з норм специфікації
3. **Реєструвати готову продукцію** — партії ГП з тарою, кількістю, прив'язкою
   до варки і ТМ
4. **Однією кнопкою генерувати паспорт якості (PDF)** за затвердженим шаблоном
   `Шаблон_Паспорта_Якості` (gdoc ID `1DkJyL0A38qZNpbWdr8PkutwpvlTHKlePXcfU3XLlnQY`)

## Що цей проєкт НЕ робить

- Не веде рецептури (це окрема система)
- Не обчислює собівартість і не керує закупкою
- Не друкує етикетки, штрих-коди, упаковку
- Не управляє складом / залишками
- Не інтегрується з 1С / бухгалтерією
- Не редагує існуючий Google Sheet — лише читає (одноразово, на імпорт) і
  пише туди дзеркальну копію після кожного збереження (тільки на читання людям)

## Користувачі

Біолаб технологи і фахівець з якості. 2-5 людей одночасно. Простий
логін + пароль, без ролей у першій версії (всі бачать і редагують усе).

## Хостинг

Локально на ПК користувача (Olena), доступ для команди через локальну мережу
(`http://<ip-ПК>:8000`). База — SQLite в одному файлі. Бекап = копіювання `.db`.

У майбутньому, якщо потрібен віддалений доступ — переїзд на Render/Railway
з міграцією SQLite → Postgres (заплановано, але не зараз).

## Як групуються замовники, продукти і артикули

Три рівні. Це найважливіша частина моделі, плутати не можна.

```
Customer (ТМ-замовник)              ← клієнт-власник торгової марки
   │                                  Crumb, Plis Me, Pinky, Пухтенко, Азаренко, ANKIE002, ...
   │ 1-to-many
   ↓
ProductVariant (товарна позиція)    ← конкретна "етикетка" для цієї ТМ
   │                                  напр. Crumb продає наш RP032 як
   │                                  "Гель для душу Chery and Santal, 200 ml"
   │ many-to-1
   ↓
Product (RP-артикул)                ← наша внутрішня рецептура
                                      RP032 = "Гель для душу" (один рецепт)
```

**Чому три рівні, а не два:**

- Один RP-артикул (одна наша рецептура) може продаватись **під різними ТМ
  під різними назвами**. RP032 → у Crumb це "Chery and Santal", у Pinky
  це "Floral wood". Рецепт той самий, етикетки різні.
- Одна ТМ може мати **багато товарів** (Pinky: гель, скраб, бомбочка).
- В існуючому Sheet "Каталог TM × ГП" 624 рядки = 624 унікальних
  комбінацій `(Customer, Product, label_name)` = ProductVariant.

**Зв'язки в системі:**

- `Specification` → FK `Product` (норми на рецептуру, а не на варіант)
- `BulkBatch` → FK `Product` + FK `Specification` (варимо рецептуру, не товар)
- `FinishedBatch` → FK `BulkBatch` + FK `ProductVariant` (з варки робимо
  конкретні товарні позиції; одна варка може закрити кілька варіантів —
  10 кг бульку розливається в гель Crumb + гель Pinky одночасно)
- `QualityCertificate` → FK `FinishedBatch` (паспорт на партію ГП конкретної ТМ)

**Що з товарами без RP:** в існуючому каталозі є рядки без RP-артикула
(товари у розробці, без затвердженої рецептури). У моделі `ProductVariant.product_id`
nullable, але **паспорт згенерувати неможливо** доки не присвоєно RP і
не затверджена специфікація. UI показує таку позицію з міткою "немає RP".

## Що користувач заводить vs що рахується автоматично

Це принципово важливо: дублювання даних — головне джерело помилок у поточному
Sheet. У застосунку **кожне число вводиться рівно один раз**.

### Customer (новий клієнт)
- ✏️ Вручну: назва ТМ, контактна особа, примітки

### Product (нова рецептура)
- ✏️ Вручну: RP-код, назва, категорія, опис

### ProductVariant (нова товарна позиція)
- ✏️ Вручну: вибір ТМ, вибір RP (можна пропустити), назва-як-у-клієнта,
  фасування (об'єм + тип тари)

### Specification (нова специфікація)
- ✏️ Вручну: вибір RP, всі норми (pH мін/макс, в'язкість, густина,
  колоїд, термо, мікро), умови зберігання, термін придатності, PAO, статус
- 🤖 Авто: `spec_id` (`SPEC-RP001-v1.0`), `version` (інкремент при оновленні
  затвердженої), `created_at`, `updated_at`

### BulkBatch (нова варка)
- ✏️ Вручну: вибір RP, № партії, дата виготовлення, обсяг (кг), факт-показники
  (pH факт, в'язкість факт, густина факт, колір/запах/зовнішній вигляд,
  колоїд/термо/мікро факт), виконавець, арбітражні зразки (так/ні), примітка
- 🤖 Авто: FK на актуальну затверджену специфікацію за RP. **Норми не
  вводяться** — UI показує їх з спеки тільки для нагадування технологу
- 🤖 Авто: `verdict` (OK / OUT) — обчислюється з порівняння факт vs норма

### FinishedBatch (фасування партії ГП)
- ✏️ Вручну: вибір BulkBatch (з якої варки), вибір ProductVariant (для кого
  пакуємо), № партії ГП, дата фасування, кількість одиниць, дата введення
  аромату (якщо є), примітка
- 🤖 Авто: фіз-хім показники **не дублюються** — паспорт читає їх із BulkBatch
- 🤖 Авто: `allowed_for_shipment` = `(bulk.verdict == 'OK')`

### QualityCertificate (паспорт)
- ✏️ Вручну: натиснути кнопку "Згенерувати паспорт" на сторінці FinishedBatch.
  Опційно — `verdict_text` (вільна фраза в кінці, типу "відповідає вимогам ТУ"),
  залишається пусто за замовчуванням
- 🤖 Авто: `passport_no` = `"ПУ-" + batch_number_gp`
- 🤖 Авто: `date_today` = поточна дата
- 🤖 Авто: `date_expiry` = `bulk.date_mfg + spec.shelf_life_months`
- 🤖 Авто: всі 10 норм у таблиці паспорта — з `Specification`
- 🤖 Авто: всі 10 фактичних результатів — з `BulkBatch`
- 🤖 Авто: метадані партії (`product_name`, `customer_tm`, `volume`, `batch_gp`,
  `date_mfg`, `pao`) — з FinishedBatch + ProductVariant + Specification
- 🤖 Авто: рендер HTML → PDF, файл зберігається в `data/passports/`

### Підсумок: одне джерело правди

| Дані | Де живуть | Куди тягнуться |
|---|---|---|
| Норми pH, в'язкість, мікро, термін, PAO | `Specification` | Паспорт, UI варки (showref) |
| Факт pH, в'язкість, мікро | `BulkBatch` | Паспорт |
| ТМ, назва-у-клієнта, тара | `ProductVariant` | Паспорт, журнал ГП |
| Висновок OK/OUT | Обчислюється | UI варки, дозвіл на паспорт |
| Дата expiry, № паспорта | Обчислюється | Паспорт |

## Доменна модель

```
Customer (ТМ-замовник)
  ├─ trademark_name  (Crumb, Plis Me, Pinky, ...)
  ├─ contact_person
  └─ notes

Product (RP-артикул, наша рецептура)
  ├─ rp_code        (RP001, RP032, L301, ...)
  ├─ internal_name  (внутрішня назва, як у нашій документації)
  ├─ category       (волосся / тіло / обличчя / тощо)
  └─ description

ProductVariant (товарна позиція = ТМ × RP × етикетка)
  ├─ FK Customer
  ├─ FK Product (nullable — для товарів без присвоєного RP)
  ├─ label_name     ("Гель для душу Chery and Santal")
  └─ notes
  (тара НЕ зберігається тут — вводиться у FinishedBatch при фасуванні)

PackagingType (тип тари — невеликий довідник)
  ├─ name           (Флакон / Туба / Ролер / Банка / Бомбочка / Скраб / Баттер / Саше)
  ├─ typical_unit   (мл / г / мл або г — підказка лаборанту)
  └─ is_active      (bool — деактивувати замість видалення)

Specification (нормативи на рецептуру)
  ├─ FK Product
  ├─ spec_id        (SPEC-RP001-v1.0)
  ├─ version
  ├─ status         (затверджено / потребує верифікації / чернетка)
  ├─ appearance, color, smell   (текстові норми)
  ├─ ph_min, ph_max
  ├─ viscosity_norm (рядок: "S3V12 Cp=5000-8000 mPas")
  ├─ density_norm   (рядок: "0,9-1,0 г/мл")
  ├─ colloid_norm, thermo_norm  (зазвичай "стабільний")
  ├─ micro_total_norm, micro_yeast_norm
  ├─ storage_conditions
  ├─ shelf_life_months   (24)
  ├─ pao_months          (12)
  └─ source, notes

User (співробітник з логіном)
  ├─ email
  ├─ full_name      (для підписів і audit logs)
  ├─ role           (admin / lab / manager)
  └─ password_hash

BulkBatch (варка)
  ├─ FK Product, FK Specification (поточна затверджена)
  ├─ batch_number   ("RP21/05/26(1)" — формат RP+ДД/ММ/РР(N))
  ├─ date_mfg
  ├─ volume_kg
  ├─ appearance_fact, color_fact, smell_fact
  ├─ ph_fact
  ├─ viscosity_fact, viscosity_unit
  ├─ density_fact
  ├─ colloid_fact, thermo_fact
  ├─ micro_total_fact, micro_yeast_fact
  ├─ operator (виконавець)
  ├─ arbitrary_samples_taken (bool)
  ├─ micro_test_started_at  (дата постановки на мікробіологію, зазвичай = date_mfg)
  ├─ micro_completed_at     (дата завершення; nullable — null поки результат не готовий)
  ├─ verdict        (OK / OUT / PENDING — обчислюється з норм vs факт + статус мікро)
  ├─ notes
  └─ created_at, updated_at, created_by_user_id, updated_by_user_id

FinishedBatch (партія ГП — фасування варки під конкретний варіант)
  ├─ FK BulkBatch        (звідки взято фіз-хім)
  ├─ FK ProductVariant   (для кого пакуємо — ТМ + label)
  ├─ FK PackagingType    (тип тари з довідника — Флакон / Туба / ...)
  ├─ packaging_volume    (цифра обсягу/ваги одиниці тари — 200, 50, 100, ...)
  ├─ packaging_unit      (одиниця: мл або г)
  ├─ batch_number_gp     ("21/05/26(01)" — формат ДД/ММ/РР(NN), БЕЗ префікса RP)
  ├─ date_filling
  ├─ date_aroma_added    (опційно — дата введення аромату в ангро)
  ├─ quantity_units      (шт)
  ├─ volume_total_kg     (загальний обсяг партії ГП у кг, вводиться руками)
  ├─ filling_operator    (виконавець фасування, з довідника)
  ├─ allowed_for_shipment (bool — автоматично якщо bulk.verdict=OK)
  └─ notes

QualityCertificate (паспорт)
  ├─ FK FinishedBatch
  ├─ passport_no    ("ПУ-21/05/26(01)")
  ├─ generated_at
  ├─ pdf_path       (data/passports/ПУ-21-05-26-01.pdf)
  ├─ status         (створено / відкликано)
  └─ verdict_text   (з шаблону, додаткова фраза)
```

## Залізні правила обробки даних

1. **Висновок OK/OUT/PENDING — автоматично, не вручну.** Три стани:
   - `PENDING` — мікробіологія ще не завершена (`micro_completed_at IS NULL`).
     Це нормально 3-7 днів після варки. Партію ГП **можна створити**, але
     паспорт **згенерувати не можна** до завершення мікро.
   - `OUT` — хоч один показник за межами норми. Партію ГП не створюємо,
     паспорт не генеруємо.
   - `OK` — мікро завершена І всі показники в нормі. Тільки тоді паспорт
     можна згенерувати.

   Числове порівняння — тільки pH (`ph_min <= ph_fact <= ph_max`). Текстові
   (колоїдна, термо, мікро total, мікро yeast) — case-insensitive рівність.
   В'язкість і густина (норми рядкові типу "S3V12 Cp=5000-8000 mPas") не
   парсимо числово — пропускаємо в автоматичній перевірці.

2. **Не редагуємо затверджені специфікації.** Якщо специфікація має статус
   `затверджено`, її не можна змінити — створюється нова версія
   (`SPEC-RP001-v1.1`) зі статусом `чернетка`. Перехід на нову — окрема дія
   (перепризначення FK на BulkBatch).

3. **Не видаляємо нічого.** Партії та паспорти можна тільки **відкликати**
   (soft delete з причиною). У БД лишається запис, у UI показано викреслене.

4. **Паспорт генерується тільки якщо bulk verdict = OK.** Якщо варка має OUT —
   паспорт **взагалі не можна згенерувати**. Партію ГП теж не можна створити з
   OUT-варки. Технолог має повернутися у журнал варок, перевірити показники
   (можливо помилка вводу), виправити дані варки. Тільки коли verdict стане OK
   — партія ГП і паспорт стають доступні. Це жорстке правило, без "обходу".

   **Виняток для імпортованих історичних даних:** старі паспорти з реєстру
   (до запуску застосунку) імпортуються **як архів без перевірки висновку** —
   ставимо `legacy_import=True` і не показуємо їм бейдж verdict. Це через те,
   що в попередньому Sheet формули рвалися і деякі паспорти отримали OUT
   помилково. Для **нових варок** правило діє в повну силу.

5. **⚠ GMP-вимога: паралельне Excel/Sheet дзеркало завжди.** Журнал якості
   є СОП-документом під GMP — втрата записів = serious compliance issue
   (можуть зняти сертифікат, аудит провалити). Тому **три рівні захисту даних**:

   1. **SQLite/Postgres** — основне сховище в застосунку. ACID-транзакції,
      бекап `.db` файлу щодня (Windows Task Scheduler або Railway snapshots).
   2. **Google Sheet — live mirror, синхронно при кожному збереженні**
      (НЕ cron, НЕ batch). Кожен `INSERT/UPDATE` у БД одразу пише копію в
      окремий Sheet через service account. Якщо застосунок впаде — всі
      записи в Sheet залишаються живими, можна відкрити для звіту.
   3. **XLSX експорт за запитом** — кнопка "Експортувати журнал у XLSX"
      на кожній сторінці списку (варки, ГП, паспорти, специфікації). Звіт
      для аудитора або просто snapshot на конкретну дату.

   Реалізація mirror: окремий Google Sheet (новий, не той що джерело
   імпорту) — кожна вкладка відповідає одній таблиці БД. Service account
   пише туди. Люди не редагують Sheet — read-only через Drive permissions.
   Якщо Sheet API недоступний — застосунок чекає, потім ретраїть (не
   приймає запис тільки в БД без mirror). Це гарантія GMP-цілісності.

6. **Назви, артикули, ТМ, виконавці, тип тари — тільки з довідників.**
   Ніяких вільних text-input полів для цих сутностей. Якщо потрібного запису
   нема в довіднику — спочатку додати в довідник, потім використовувати.
   Це правило діє для:
   - Продукт (RP) → з `Product`
   - ТМ-замовник → з `Customer`
   - Назва товару (label) → з `ProductVariant`
   - Тип тари → з `PackagingType` (Флакон/Туба/Ролер/Банка/Бомбочка/...)
   - Виконавець (варки і фасування) → з довідника працівників (`User`)
   - Специфікація → з `Specification` (поточна затверджена для цього RP)

   Тільки **обсяг тари** (цифра в мл або г) вводиться руками — бо одна
   і та сама ТМ-позиція може фасуватись у різні об'єми.

   Вільний текст дозволено лише в полях `notes`/`description`/`verdict_text`.

7. **Ролі і авторизація — обов'язкові з v1.** Кожна дія прив'язана до
   `User` через логін/пароль. Три ролі:

   | Роль | Може робити | Не може |
   |---|---|---|
   | **admin** (Olena) | Все: довідники, специфікації, варки, ГП, паспорти, користувачі | — |
   | **lab** (лаборант) | Вносити варки, мікро-результати. Створювати партії ГП. Дивитись все. | Редагувати довідники, специфікації, ТМ, продукти, variants |
   | **manager** | Генерувати/відкликати паспорти. Дивитись все. | Створювати варки, редагувати довідники |

   Перевірка ролей на бекенді (FastAPI dependency), не в UI. UI може ховати
   кнопки, але кожен endpoint перевіряє роль користувача.

8. **Аудит — хто/коли/що.** У кожному записі: `created_by_user_id`,
   `updated_by_user_id`, `created_at`, `updated_at`. Кожна **зміна** (UPDATE)
   додатково пишеться у таблицю `audit_log`: `user_id`, `entity_type`,
   `entity_id`, `field`, `old_value`, `new_value`, `timestamp`. Запис в
   `audit_log` ніколи не видаляється і не змінюється — append-only.

9. **Мікробіологія — окрема стадія, не блокує введення інших показників.**
   День 0: технолог зварив, ввів органолептику + pH + в'язкість + густину +
   колоїдну + термо. Поставив пробу на мікробіологію (`micro_test_started_at`
   = сьогодні). Висновок поки PENDING.

   Через 3-7 днів той самий або інший лаборант відкриває варку, редагує:
   вводить `micro_total_fact`, `micro_yeast_fact` і ставить `micro_completed_at`.
   Висновок автоматично перераховується → OK або OUT.

## Правила безпеки

1. **Бекапи `.db` щодня.** Скрипт `scripts/backup_db.py` копіює
   `data/journal.db` у `data/backups/journal-YYYY-MM-DD.db` (cron / Windows
   Task Scheduler).

2. **Soft delete завжди.** Жоден API не робить `DELETE FROM`. Тільки
   `is_deleted=TRUE` + причина + `deleted_at`.

3. **Credentials НЕ в git.** `service_account.json` і `.env` у `.gitignore`.

4. **Логування дій.** Кожне створення/редагування пишеться в `data/audit.log`:
   `{timestamp} {user} {action} {entity_id} {old_value} → {new_value}`.

5. **Перед першим імпортом — снапшот існуючого Sheet** у
   `data/imported_snapshot_YYYY-MM-DD.csv` (всі 9 вкладок). Якщо щось піде
   не так — є з чого відновити.
